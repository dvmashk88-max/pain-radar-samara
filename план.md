## Короткий вывод

Текущая версия бота технически работает, но как бизнес-радар пока невалидна.

Проблема не в Telegram-боте, SQLite или AI-анализе. Проблема в источниках данных: бесплатные публичные источники, которые сейчас подключены, почти не дают качественных пользовательских жалоб по нишам и городам.

Текущий результат нельзя интерпретировать как "болей нет". Правильная интерпретация: "бот не подключён к местам, где эти боли реально живут".

## Почему текущая версия проваливается по данным

### 1. Telegram-каналы дают не тот тип контента

Подключённые публичные каналы Самары, Тольятти, Новокуйбышевска и Сызрани в основном публикуют:

- городские новости;
- ДТП;
- криминал;
- пожары;
- объявления;
- рекламу;
- официальную повестку.

Это не системный поток клиентских жалоб на бизнес.

После фильтрации мусора остаётся очень мало сигналов. Это нормально для таких источников.

### 2. DuckDuckGo нестабилен

DuckDuckGo HTML с Railway часто отдаёт `HTTP 202` почти на все запросы.

Даже когда он отвечает, он возвращает заголовки поисковой выдачи, а не сами отзывы. Это слабый сигнал. Он может быть полезен только как вспомогательный источник, но не как основа продукта.

### 3. Старые сильные источники отключены

Ранее были отключены:

- Яндекс;
- Авито;
- Отзовик.

Причины:

- CAPTCHA;
- антибот;
- HTTP 429;
- HTTP 400;
- HTTP 404/507.

Именно такие источники ближе к настоящим болям, но они плохо парсятся с Railway без отдельной инфраструктуры.

### 4. Классификация ниш была слишком грубой

Были ложные срабатывания по нише "автосервис", в том числе из-за коротких подстрок. Это частично исправлено, но полноценная классификация требует другого подхода:

- регулярки с границами слов;
- словари по нишам;
- ручная валидация;
- AI-классификация только после первичного отбора.

### 5. SQLite на Railway хранит тестовый мусор

Первые сканы сохраняли мусорные и перекошенные данные. Поэтому `/stats` может показывать старую картину даже после исправления фильтров.

Для тестов добавлена `/reset_data`.

## Что считать настоящей валидацией идеи

Идея считается живой только если бот регулярно находит:

- 20-50 свежих релевантных жалоб в неделю;
- по нескольким городам, а не только по Самаре;
- по нескольким нишам, а не по одной;
- с понятным текстом боли;
- со ссылкой на источник;
- без новостей, ДТП, криминала и рекламы.

Минимальный хороший сигнал:

```text
Город: Тольятти
Ниша: стоматология
Боль: нет записи / дорого / хамство
Текст: реальный отзыв клиента
Источник: ссылка на отзыв или обсуждение
```

Если таких сигналов нет, AI-анализ бессмысленен.

## Стратегия спасения проекта

### Направление 1. Перейти от новостных каналов к отзывам

Главный приоритет: подключить источники отзывов, а не городские новости.

Потенциальные источники:

- 2ГИС;
- Яндекс Карты;
- Flamp;
- Zoon;
- ProDoctorov;
- Yell;
- Google Maps, если доступен;
- тематические сайты по клиникам, ремонту, авто, доставке.

Риск: многие источники защищены от парсинга. Нужна отдельная проверка по каждому.

### Направление 2. Сделать полуручной MVP

Вместо сканирования всего интернета:

1. Выбрать 3 города:
   - Самара;
   - Тольятти;
   - Новокуйбышевск или Сызрань.

2. Выбрать 5 ниш:
   - стоматологии;
   - клиники;
   - автосервисы;
   - ремонт квартир;
   - ЖКХ/управляющие компании.

3. Собрать список конкретных компаний вручную или полуавтоматически.

4. По каждой компании искать отзывы.

Плюс такого подхода: меньше шума, проще проверить бизнес-ценность.

### Направление 3. Сделать режим "исследование ниши"

Вместо общего `/scan` добавить команды:

```text
/scan_niche стоматология Самара
/scan_niche автосервис Тольятти
/scan_niche жкх Новокуйбышевск
```

Бот должен искать не абстрактные "боли региона", а конкретную нишу в конкретном городе.

Это ближе к продукту, который можно продавать предпринимателю.

### Направление 4. Добавить просмотр сырых сигналов

Нужна команда:

```text
/last
```

Она должна показывать последние 10 сохранённых сигналов:

- город;
- ниша;
- боль;
- источник;
- текст;
- ссылка.

Без `/last` невозможно быстро понять, бот нашёл реальную боль или мусор.

### Направление 5. Добавить ручную разметку качества

Команды:

```text
/good <id>
/bad <id>
```

Или inline-кнопки под сигналом:

- `👍 Боль`;
- `👎 Мусор`.

Это даст простую обучающую выборку и поможет улучшать фильтры.

## План действий

### Этап 1. Зафиксировать текущий статус

Цель: признать, что текущий общий радар невалиден по источникам.

Действия:

- оставить текущий бот как технический каркас;
- не делать выводы о рынке по текущей статистике;
- очистить тестовую базу через `/reset_data`;
- добавить `/last`, чтобы смотреть сырые данные.

Результат этапа: понятно, что именно бот нашёл, а не только агрегаты.

### Этап 2. Проверить источники отзывов вручную

Цель: найти хотя бы 1-2 источника, где реально есть поток жалоб.

Действия:

- вручную открыть 2ГИС, Zoon, ProDoctorov, Flamp, Яндекс Карты;
- для каждой ниши и города проверить, есть ли свежие отзывы;
- записать:
  - URL;
  - доступность без авторизации;
  - есть ли даты отзывов;
  - есть ли текст отзывов в HTML;
  - нужна ли JS-отрисовка;
  - есть ли антибот.

Результат этапа: таблица источников с вердиктом "можно/нельзя/сложно".

### Этап 3. Сделать один качественный источник

Цель: лучше один нормальный источник, чем пять мусорных.

Действия:

- выбрать самый доступный источник отзывов;
- написать парсер только под него;
- сохранять реальные отзывы;
- добавить дату отзыва, если доступна;
- фильтровать отзывы по городам и нишам.

Результат этапа: 20-50 реальных отзывов вместо 1-5 случайных заголовков.

### Этап 4. Перейти к нишевым сканам

Цель: сделать бота полезным для конкретного предпринимателя.

Действия:

- добавить команду `/scan_niche`;
- ограничить поиск городом и нишей;
- выводить:
  - топ болей;
  - примеры отзывов;
  - частотность;
  - возможные продуктовые идеи.

Результат этапа: бот отвечает на вопрос "что болит у клиентов стоматологий в Тольятти?", а не "что вообще болит в регионе?".

### Этап 5. Улучшить хранение

Цель: перестать терять данные на Railway.

Действия:

- перейти с SQLite на Railway PostgreSQL или Supabase;
- добавить миграции;
- хранить:
  - source;
  - city;
  - niche;
  - company;
  - review_date;
  - collected_at;
  - text;
  - url;
  - quality_label.

Результат этапа: можно копить историю месяцами.

### Этап 6. Проверить коммерческий смысл

Цель: понять, можно ли это продавать.

Действия:

- выбрать одну нишу, например стоматологии;
- собрать 100-200 отзывов по Самаре и Тольятти;
- сделать отчёт:
  - топ-10 болей;
  - примеры;
  - частотность по городам;
  - идеи улучшения сервиса;
  - рекламные гипотезы.

Проверка:

- показать отчёт 3-5 предпринимателям;
- спросить, заплатили бы они за такой мониторинг;
- если никто не видит ценности, проект закрыть или развернуть.

## Критерии продолжения

Продолжать проект стоит, если после подключения нормального источника:

- минимум 60% найденных сигналов релевантны;
- находится не меньше 20 сигналов в неделю;
- есть разные ниши и города;
- сырые примеры выглядят как реальные клиентские боли;
- отчёт можно показать предпринимателю без стыда.

Закрывать или радикально менять идею стоит, если:

- источники не дают данных;
- парсинг требует тяжёлой инфраструктуры;
- качество ниже 30%;
- предприниматели не видят ценности в отчётах.
